Build the initial foundation for a school management application called King's Class, inspired by ClassDojo.

Use this stack:

* React

* TypeScript

* Express.js

* Prisma

* PostgreSQL

This is a client-facing prototype, so the frontend quality matters a lot. The UI should look polished, intentional, and strongly inspired by the school's own visual identity. Use the attached images as the main visual reference for colors, mood, and overall style. Do not create a generic admin dashboard.

Implement only:

* Login page

* Real backend authentication

* One seeded Admin account

* Admin layout after login

* Basic navigation tabs for future sections

* Class Feed section

Requirements:

* No sign-up page.

* Only the Admin account needs to exist right now.

* After login, redirect to the Admin area.

* The Admin area should have clean navigation tabs, but do not fill unfinished sections with fake feature content or useless placeholder slop.

* Implement a Class Feed section inspired by social-media-style timelines.

* The feed should display realistic dummy posts and demonstrate how teachers would publish updates to a class.

* Focus on creating a polished frontend experience and realistic UI interactions.

* Do not implement a complete backend for posts, persistence, editing, deletion, or moderation at this stage.

* Use mock or dummy data where appropriate.

* Structure the frontend so that a real backend can be connected later without requiring major changes to the UI architecture.

* Keep the backend simple and expandable.

* Keep the frontend clean, reusable, and easy to build on later.

Focus on making this first version feel like a polished foundation for King's Class, not a complete product.

Follow good project architecture and avoid monolithic files.

* Use separate `frontend` and `backend` folders.

* Split code into logical modules, components, pages, routes, services, and utilities where appropriate.

* Do not place large amounts of functionality, styling, or business logic into a single file.

* Create and configure a proper `.gitignore` file that excludes all generated files, dependencies, build outputs, environment files, logs, and other files that should not be committed to Git.

* Favor maintainability and scalability over quick shortcuts.
